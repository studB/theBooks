# S3 워크스페이스 지원

워크스페이스를 로컬 폴더 외에 S3 버킷/프리픽스로도 지정할 수 있게 한다. 열 때 1회 자동 pull, 그리고 수동 동기화 버튼으로 양방향 sync.

## Version

- Current: 0.7.1
- Bump: minor
- Target: 0.8.0

## Purpose

작가가 여러 기기에서 같은 책을 이어 쓰려면 워크스페이스가 클라우드에 있어야 한다. 가장 표준적인 S3를 워크스페이스로 쓸 수 있게 해서 PC/노트북/태블릿 간 자료 공유의 1차 경로를 만든다. v1은 단일 사용자가 기기를 바꿔가며 작업하는 시나리오에 한정한다.

## Scope

**In scope**
- 워크스페이스 종류 두 가지: 기존 로컬 폴더, 신규 S3 prefix (`s3://bucket/path/`)
- S3 자격증명(access key, secret, region, bucket, prefix)을 `config.json`에 저장
- 워크스페이스 피커를 모달로 확장: "로컬 폴더" / "S3" 두 옵션 → S3는 폼으로 입력
- 로컬 캐시 디렉토리: `app_data_dir()/s3-cache/<hash>/`에 미러
- 워크스페이스 활성화 시 기존 fs 명령들은 모두 로컬 캐시 경로로 동작 (수정 없음)
- 단일 `sync_workspace` 명령으로 양방향 동기화(S3↔로컬), mtime 비교로 최신 승
- 워크스페이스 처음 마운트 시 자동 pull 1회 (실패해도 앱 진입은 가능)
- 헤더에 "동기화" 버튼 + 진행 중 텍스트 표시
- 로컬 워크스페이스로 다시 전환 가능 — `set_workspace`가 S3 설정도 비워야 함

**Out of scope (later)**
- S3 호환 스토리지(R2, MinIO 등) custom endpoint
- 자격증명 안전 저장(keychain/OS credential vault)
- 충돌 해소 UI (3-way merge, diff)
- 멀티 사용자 동시 편집, 잠금
- 파일별 진행률, 스피너/프로그레스 바
- 자동 sync(쓰기마다 push, watcher)
- 마이그레이션(기존 로컬 → S3 복사)
- 자격증명 사전 유효성 검증 (실제 호출 실패 시 메시지만)
- 다른 클라우드(GCS, Dropbox, Drive)
- 휴지통 처리: S3에서는 삭제 = 객체 delete (휴지통 없음). 로컬 캐시에서는 기존대로 trash

## Stages

1. **AWS SDK + config 확장** — `Cargo.toml`에 `aws-sdk-s3`, `aws-config`, `aws-credential-types`, `tokio` (이미 있음 추정) 추가. `chat::AppConfig`에 `s3_workspace: Option<S3WorkspaceConfig>` 필드 추가 (bucket, prefix, access_key, secret_key, region). `lib.rs`에 신규 명령 등록.
2. **로컬 캐시 경로 라우팅** — `fs::workspace_root()`를 변경: S3 설정이 있으면 `app_data_dir/s3-cache/<blake3 hex 16자>/`을 반환하고 디렉토리 생성. 로컬 설정이 있으면 기존 동작.
3. **`sync_workspace` 명령** — async tauri 명령. S3 워크스페이스가 활성일 때만 동작. S3 목록(list_objects_v2 페이지네이션) + 로컬 walkdir 비교 → 각 키에 대해 pull/push/skip 결정. mtime이 같으면 skip. 결과는 `{ pulled, pushed, skipped, failed }` 카운트로 반환.
4. **`set_s3_workspace` / 피커 모달 UI** — 신규 명령 `set_s3_workspace`는 폼 데이터를 받아 config 저장하고 `workspace`(로컬)는 None으로 비운다. 프론트는 새로운 모달 컴포넌트(`WorkspacePickerModal.jsx`)에서 "로컬 폴더" / "S3" 라디오 + 조건부 폼. 기존 OS 다이얼로그 호출은 모달 내부에서.
5. **자동 1회 pull + 수동 sync 버튼** — AppShell 마운트 시 S3 워크스페이스면 `sync_workspace` 한 번 호출(실패 토스트, 진입 차단 안 함). 헤더/사이드바에 "동기화" 버튼 추가, 클릭 시 호출. 진행 중 상태 텍스트.
6. **검증** — `cargo check` + `bun run build` 통과. 실제 S3 동작은 사용자가 자기 자격증명으로 dev에서 확인.

## Constraints

- AWS SDK 첫 빌드가 매우 길다 — 의존성 늘어남을 감수
- 자격증명은 config.json 평문 (보안은 별도 plan)
- 충돌은 newest-mtime-wins 단순 규칙 — UI에 명시 안내
- 네트워크 의존 — 오프라인 시 로컬 캐시에서만 동작, sync 호출만 실패
- 모든 파일을 한 번에 받음(대용량/대량은 v1 아님)
- 기존 chat / 로컬 워크스페이스 동작 회귀 없음
- S3 키 매핑: 로컬 상대 경로 = `<prefix>/<rel_path>`, prefix가 비면 그대로
- 워크스페이스 캐시는 S3 URL이 같으면 재사용(여러 기기 간 동일한 캐시 키)

## Direction

- Crates: `aws-sdk-s3 = "1"`, `aws-config = "1"`, `aws-credential-types = "1"`, `blake3 = "1"`, `walkdir = "2"`. Tauri는 이미 tokio runtime을 제공.
- `S3WorkspaceConfig`는 `Serialize + Deserialize`, region은 String, prefix는 마지막 슬래시 정규화.
- 클라이언트는 호출마다 새로 만든다(v1 단순). credentials는 `Credentials::new(access_key, secret_key, None, None, "static")`로 직접 주입.
- `workspace_root()` 캐시 경로 키: `blake3(format!("s3://{bucket}/{prefix}")).to_hex()[..16]`. 캐시 디렉토리는 첫 호출 시 lazy 생성.
- `sync_workspace`는 단순한 한 패스 알고리즘: (1) S3 목록을 수집(`HashMap<key, (size, last_modified)>`), (2) 로컬 파일들 수집(`HashMap<rel_path, mtime>`), (3) 합집합 순회하며 결정. 삭제 동기화는 v1 미지원(어느 쪽이든 삭제했다면 sync에서 다시 복원될 수 있음 — 한계로 명시).
- `set_s3_workspace` 호출 후 프론트는 곧바로 `sync_workspace` 호출 + `list_workspace` 새로고침.
- 프론트 모달: 기존 인앱 모달 패턴 부재 → 새 컴포넌트(`WorkspacePickerModal.jsx`)에 라디오 + 조건부 폼. 닫기/취소 핸들링.
- 진행 상태는 React state 하나(`syncing: boolean`)와 메시지 한 줄로 충분. 자세한 진행률은 v1 아님.
